**The key insight:** `handleFilterSelect` in EventListener currently does two things: (1) toggles the filter in the URL, and (2) the URL change triggers the URL-driven $effect which enqueues a FILTER event. The new pattern replaces step 1: components write directly to `searchManager.selectedFilters`, and a NEW reactive $effect in EventListener watches `searchManager.selectedFilters` and enqueues FILTER events + updates the URL as a side-effect.

**In `EventListener.svelte`:**

1. **Delete** the `handleFilterSelect` function entirely.

2. **Delete** the `$effect` that registers `uiCtx.handleFilterSelect = handleFilterSelect`. After Plan 03 removed `uiCtx.applySort`, this `$effect` only had `handleFilterSelect`. Delete the entire `$effect` block.

3. **Add a new reactive $effect** that watches `searchManager.selectedFilters` and enqueues FILTER events. This replaces the URL-driven approach for filter changes:

```typescript
// Watch searchManager filter state — components write here directly,
// EventListener reacts by enqueuing FILTER event and updating URL as side-effect
let lastFiltersJson = $state(JSON.stringify(searchManager.selectedFilters));

$effect(() => {
  const currentFilters = searchManager.selectedFilters;
  const currentJson = JSON.stringify(currentFilters);

  if (currentJson === lastFiltersJson) return;  // no change
  lastFiltersJson = currentJson;

  const q = searchManager.inputValue || '';
  const view = untrack(() => viewCtx.activeView);

  // Update URL as side-effect (not source of truth)
  untrack(() => {
    updateSearchUrl({ q, filters: currentFilters, view });
  });

  // Enqueue filter event
  if (q || currentFilters.length > 0) {
    queue.enqueue({ type: 'FILTER', q, filters: currentFilters, view });
  } else {
    queue.enqueue({ type: 'FILTER', q: '', filters: [], view });
  }
});
```

4. **Update the URL-driven search $effect** — it should still handle view changes from URL (popstate/back-forward) and search queries from the Toolbar, but filter changes now come through the searchManager watcher above. The URL $effect should NOT enqueue FILTER events when the change was caused by a filter selection (the searchManager watcher handles that).

To prevent double-enqueue: the URL $effect currently syncs `searchManager.selectedFilters` from URL params. After this change, the direction reverses — searchManager drives URL. The URL $effect should still handle:
- View changes (VIEW_CHANGE event)
- Search query changes from Toolbar (FILTER event — Toolbar writes to URL, not searchManager)
- Initial page load skip

The tricky part: avoid the circular loop where searchManager changes -> URL updates -> URL $effect fires -> tries to enqueue again. The URL $effect already `untrack`s its writes to searchManager. But it reads `reactiveUrl.searchParams.getAll('filter')` which triggers on URL changes.

**Simplest approach:** Keep the URL $effect mostly as-is for VIEW_CHANGE and search query. For the filter path, add a guard: if the URL filter params match `searchManager.selectedFilters`, skip enqueue (the searchManager watcher already handled it).

Alternatively: the URL $effect can focus solely on VIEW_CHANGE and text search (q param). Filter events are exclusively handled by the searchManager watcher. Remove the filter-related enqueue logic from the URL $effect.

**This is complex wiring — use executor discretion** for the exact implementation. The key constraints are:
- Components write to `searchManager.selectedFilters` directly
- EventListener watches searchManager and enqueues FILTER events
- URL is updated as side-effect, not source of truth
- No double-enqueue for the same filter change
- Popstate/back-forward for view changes still works via URL $effect
- Toolbar search (q param) still works via URL $effect
