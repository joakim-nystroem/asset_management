**Change 3 ‚Äî handleAddNewRow scroll target (lines 381-383):**
After adding a new row, the scroll target should be the bottom of the content area. Since `assets` no longer includes new rows, `assets.length - 1` would scroll to the last real row, not the new row.

Fix: After `rowGen.addNewRows(1, template)` and `await tick()`, instead of using `assets.length - 1`, use a larger index that accounts for new rows:

```typescript
// BEFORE:
const lastRowIndex = assets.length - 1;
viewCtx.scrollToRow = lastRowIndex;
selection.selectCell(lastRowIndex, 1);

// AFTER:
// Scroll to the last new row position (filteredAssets count + new row count - 1)
const lastNewRowIndex = filteredAssets.length + rowGen.newRowCount - 1;
viewCtx.scrollToRow = lastNewRowIndex;
selection.selectCell(lastNewRowIndex, 1);
```

Note: `viewCtx.scrollToRow` triggers `virtualScroll.ensureVisible()` in GridContainer. For indices beyond the virtual scroll range, ensureVisible will scroll to the bottom of the container. The RowGeneration component is rendered after the virtual-scroll chunk, so scrolling to the bottom will make new rows visible.

However, `virtualScroll.ensureVisible()` may not know about new row positions. A simpler approach: after adding a new row, directly scroll the container to the bottom. But we don't have access to the scrollContainer ref here.

Alternative: Keep the scroll approach but add a check. If `lastNewRowIndex >= filteredAssets.length`, the virtual scroll will clamp it. Instead, set `viewCtx.scrollToRow` to `filteredAssets.length - 1` (last real row) ‚Äî this scrolls near the bottom, and RowGeneration's rows are immediately after, so they'll be visible. Then select the new row cell:

```typescript
// Scroll to last real row (which positions viewport near new rows)
viewCtx.scrollToRow = filteredAssets.length - 1;
// Select the new row's first editable cell
selection.selectCell(filteredAssets.length + rowGen.newRowCount - 1, 1);
```

Actually, the cleanest approach: keep `viewCtx.scrollToRow = lastNewRowIndex` as-is. The `ensureVisible` function in GridContainer will try to scroll to that row index. Even though the virtual scroll doesn't track new row offsets, it will scroll as far down as possible ‚Äî which puts RowGeneration's new rows in view since they're rendered at the bottom of the scroll area.

Go with:
```typescript
const lastNewRowIndex = filteredAssets.length + rowGen.newRowCount - 1;
viewCtx.scrollToRow = lastNewRowIndex;
selection.selectCell(lastNewRowIndex, 1);
```

**Change 4 ‚Äî navigateToError (lines 317-353):**
This function already handles new rows separately by iterating `rowGen.newRows` and computing indices as `filteredAssets.length + i`. This logic is CORRECT and should be KEPT as-is. The `invalidCells` array includes both existing row errors (from `changes.getInvalidCellKeys()`) and new row errors (from `rowGen.isNewRowFieldInvalid`). The indices for new rows use `filteredAssets.length + i`, which is the same coordinate system RowGeneration uses for `actualIndex`.

The `selection.selectCell(target.row, target.col)` and `viewCtx.scrollToRow = target.row` will work for new row indices because:
- `selectCell` just updates `selectionStart`/`selectionEnd` in context ‚Äî it doesn't index into assets
- `scrollToRow` will scroll to the bottom where RowGeneration renders

**No change needed for navigateToError.** It already works correctly with the new architecture.

**Also verify:** The `errorNavigationIndex` and dirty cell tracking `$effect` (lines 134-155) use `assets` which is now only filteredAssets. The `assetIdMap` is built from `assets` ‚Äî since new rows are no longer in `assets`, their IDs won't appear in the map. This is correct because dirty cell tracking for new rows is handled by RowGeneration, not by EventListener's `$effect`.
