Create `frontend/src/lib/grid/components/row-generation/RowGeneration.svelte` — a self-contained component that renders new rows and handles all new-row interactions.

**Script block setup:**
- Import and read contexts: `getRowGenControllerContext`, `getColumnContext`, `getEditingContext`, `getDataContext`, `getViewContext`, `getUiContext`
- Import `GridRow` from `$lib/components/grid/GridRow.svelte`
- Import `FloatingEditor` from `$lib/grid/components/floating-editor/FloatingEditor.svelte`
- Import `createEditController` from `$lib/grid/utils/gridEdit.svelte.ts`
- Import `createColumnController` from `$lib/grid/utils/gridColumns.svelte.ts`
- Import `createRowController` from `$lib/grid/utils/gridRows.svelte.ts`
- Import `toastState` from `$lib/components/toast/toastState.svelte`
- Create local controller instances: `edit = createEditController()`, `columns = createColumnController()`, `rows = createRowController()`
- Read rowGen controller from context: `const rowGen = getRowGenControllerContext()`

**Important context lifecycle note:** The rowGen controller is created and published by `GridContextProvider.svelte` (the common ancestor). RowGeneration does NOT create or publish the controller — it reads it from context via `getRowGenControllerContext()`. This is safe because GridContextProvider initializes before GridContainer (and therefore before RowGeneration).

**CONTEXT.md deviation note:** CONTEXT.md lists "Remove `createRowGenerationController()` from GridContextProvider" as a cleanup requirement. This instruction is **superseded** by the architecture research (RESEARCH.md Pitfall 1 / Option C). Due to Svelte 5 context lifecycle constraints, `setRowGenControllerContext()` MUST be called during synchronous initialization of a common ancestor that initializes BEFORE consumers (EventListener, Toolbar). GridContextProvider is that ancestor. If the controller were created in RowGeneration instead, EventListener and Toolbar would crash with "context not found" because they initialize before GridContainer (and therefore before RowGeneration). **GridContextProvider MUST continue creating and publishing the rowGen controller.** Do NOT remove `createRowGenerationController()` or `setRowGenControllerContext()` from GridContextProvider.

**Template — new row rendering:**
Wrap the entire section in a container div with `position: relative` (needed for FloatingEditor positioning):

```svelte
{#if rowGen.newRows.length > 0}
  <div class="relative">
    {#each rowGen.newRows as newRow, i}
      {@const actualIndex = dataCtx.filteredAssetsCount + i}
      <div
        class="flex border-b border-neutral-200 dark:border-slate-700 bg-blue-50/70 dark:bg-blue-500/10 border-l-4 border-l-blue-400"
        style="height: {rows.getHeight(actualIndex)}px;"
        onmousedown|stopPropagation={(e) => handleMouseDown(e, actualIndex, i)}
        ondblclick|stopPropagation={(e) => handleDblClick(e, actualIndex)}
        oncontextmenu|stopPropagation={(e) => handleContextMenu(e, i)}
      >
        <GridRow asset={newRow} keys={colCtx.keys} {actualIndex} />
      </div>
    {/each}

    <!-- FloatingEditor for new rows only -->
    {#if editCtx.isEditing && editCtx.editRow >= dataCtx.filteredAssetsCount}
      <FloatingEditor onSave={(change) => {
        const newRowIndex = editCtx.editRow - dataCtx.filteredAssetsCount;
        rowGen.updateNewRowField(newRowIndex, change.key, change.newValue);
      }} />
    {/if}

    <!-- Inline context menu for new rows -->
    {#if showContextMenu}
      <!-- positioned fixed at contextMenuX/contextMenuY -->
    {/if}
  </div>
{/if}
```

**CRITICAL — Event propagation:** All event handlers on the new-row wrapper divs MUST call `e.stopPropagation()` to prevent GridContainer's delegated handlers from firing. GridContainer's `onmousedown`, `ondblclick`, `oncontextmenu` use `closest('[data-row][data-col]')` and then index into `dataCtx.assets[row]`. New row indices (>= filteredAssetsCount) would be out of bounds in `dataCtx.assets` since new rows are no longer mixed in. **Use Svelte's event modifier syntax: `onmousedown={(e) => { e.stopPropagation(); ... }}`** (Svelte 5 does not support `|stopPropagation` modifier — use manual `e.stopPropagation()` call).

**Event handlers:**

`handleMouseDown(e, actualIndex, newRowIndex)`:
- Call `e.stopPropagation()` first
- Parse cell from `closest('[data-row][data-col]')`
- If editing, just select cell: `selection.selectCell(actualIndex, col)` where `selection = createSelectionController()` (import it)
- Otherwise, select cell normally

`handleDblClick(e, actualIndex)`:
- Call `e.stopPropagation()` first
- Parse cell from `closest('[data-row][data-col]')`
- Guard: if `!dataCtx.user` → toast "Log in to edit."
- Guard: if key is `'id'` → toast "ID column cannot be edited."
- Call `selection.selectCell(actualIndex, col)`
- Get current value: `String(rowGen.newRows[actualIndex - dataCtx.filteredAssetsCount]?.[key] ?? '')`
- Call `edit.startEdit(actualIndex, col, key, currentValue)`

`handleContextMenu(e, newRowIndex)`:
- Call `e.stopPropagation()` and `e.preventDefault()`
- Set local state: `contextMenuVisible = true`, `contextMenuX = e.clientX`, `contextMenuY = e.clientY`, `contextMenuRowIndex = newRowIndex`

**Inline context menu:**
- Minimal menu with just "Delete Row" (styled same as main context menu)
- On click: `rowGen.deleteNewRow(contextMenuRowIndex)`, toast "New row deleted.", close menu
- Close on click-outside: add a fixed backdrop div

**FloatingEditor positioning for new rows:**
Do NOT use `computeEditorPosition()` from `floatingEditor.svelte.ts` for new rows. That function subtracts `virtualScroll.getOffsetY(rows)` (the chunk's translateY origin), which produces wrong coordinates for new rows rendered OUTSIDE the translated chunk (RESEARCH.md Pitfall 5).

**Definitive approach — local position computation:** The FloatingEditor is rendered inside the `position: relative` container of RowGeneration, with `position: absolute`. Compute position locally:
- `top = (editCtx.editRow - dataCtx.filteredAssetsCount) * rows.getHeight(editCtx.editRow)` (or accumulate heights for rows before the edit row within the new-rows section: `sum of rows.getHeight(dataCtx.filteredAssetsCount + j) for j in 0..newRowIndex-1`)
- `left = sum of column widths before editCtx.editCol` (use `columns.getColumnOffset(editCtx.editCol)` or manually sum `columns.getWidth(j)` for `j in 0..editCol-1`)
- `width = columns.getWidth(editCtx.editCol)`
- `height = rows.getHeight(editCtx.editRow)`

Pass these computed values as props to FloatingEditor (or wrap FloatingEditor in an absolutely-positioned div with these style values). This keeps positioning self-contained within RowGeneration and avoids any dependency on the virtual-scroll coordinate system.

**Dirty cell overlays for new rows:**
Add a local `$derived` that computes dirty cell overlays for invalid new-row fields, similar to how GridOverlays does it for existing rows. Use `rowGen.isNewRowFieldInvalid(i, key)` to determine which cells need yellow/green overlays. Render these overlays inside the RowGeneration container with absolute positioning.

The overlay computation for new rows:
- Iterate visible new rows (all of them — no virtual scroll needed since there are typically few)
- For each cell: check `rowGen.isNewRowFieldInvalid(newRowIndex, key)`
- If dirty (has been edited): check if invalid → yellow overlay, else → green overlay
- Position: top = row offset within RowGeneration container, left = sum of column widths before col, width = column width, height = row height

To detect "dirty" (edited) cells for new rows, check if the value is non-empty (a new row cell that has been edited has a value). Or simply show overlays for invalid fields only (yellow) since new rows don't have a "valid change" concept — they're all pending.

Actually, for new rows, the overlay logic is simpler:
- Show yellow overlay for cells where `rowGen.isNewRowFieldInvalid(i, key)` returns true
- This only matters after validation has run (on commit attempt)
- Use `selection.dirtyCells` already set by EventListener/EventHandler for the overlay computation
