**In `GridContainer.svelte`, `handleHeaderClick` function (line 83):**

Change:
```typescript
const filterItems = searchManager.getFilterItems(key, dataCtx.assets, dataCtx.baseAssets);
```
to:
```typescript
const filterItems = searchManager.getFilterItems(key, dataCtx.baseAssets, dataCtx.baseAssets);
```

**Why:** `dataCtx.assets` includes new rows appended by `$derived([...filteredAssets, ...rowGen.newRows])`. New rows have empty strings for most fields, which pollute filter dropdowns with empty entries. Using `dataCtx.baseAssets` for the first argument ensures only committed rows contribute to filter item enumeration.

**Also in headerMenu.svelte (line 96):** Check the `getFilterItems` call inside the header menu template:
```svelte
{#each searchManager.getFilterItems(state.activeKey, assets, baseAssets)
```
Here `assets` is a prop from GridContainer. Since we're fixing GridContainer to pass `dataCtx.baseAssets` as the `assets` prop to HeaderMenu would be wrong (HeaderMenu needs the full assets list for other purposes). Instead, the fix is only in `handleHeaderClick` — that's where the filter items are computed for the toggle call. Actually, looking at the code more carefully: `handleHeaderClick` calls `uiCtx.headerMenu?.toggle(e, key, filterItems, isLast)` which stores `filterItems` in the header menu state. But the template in `headerMenu.svelte` line 96 calls `searchManager.getFilterItems(state.activeKey, assets, baseAssets)` directly — it does NOT use the stored filterItems.

So the fix must also be at the `<HeaderMenu>` prop site in GridContainer (line 100):

<HeaderMenu
