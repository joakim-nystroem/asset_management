**Problem:** `createChangeController()` and `createHistoryController()` are factory functions that create independent instances with local `$state`. GridOverlays creates Instance A (receives all edits), DataController creates Instance B (always empty). Commit/discard operate on Instance B and silently do nothing.

**Fix — add controller context types and share instances:**

1. In `frontend/src/lib/context/gridContext.svelte.ts`:
   - Import `ChangeController` from `$lib/grid/utils/gridChanges.svelte.ts`
   - Import `HistoryController` from `$lib/grid/utils/gridHistory.svelte.ts`
   - Add two new context pairs at the bottom (after the existing 11 pairs):
     ```typescript
     export const [getChangeControllerContext, setChangeControllerContext] = createContext<ChangeController>();
     export const [getHistoryControllerContext, setHistoryControllerContext] = createContext<HistoryController>();
     ```
   - These hold the controller INSTANCES (with their local $state), unlike the existing `ChangeContext` which only holds boolean flags.

2. In `frontend/src/lib/components/grid/GridOverlays.svelte`:
   - Import `setChangeControllerContext` and `setHistoryControllerContext` from gridContext.svelte.ts
   - After the existing `const changes = createChangeController()` and `const history = createHistoryController()` lines, add:
     ```typescript
     setChangeControllerContext(changes);
     setHistoryControllerContext(history);
     ```
   - This publishes the SAME instances that receive edits (via `changes.update()` and `history.record()`) into context so DataController can read them.

3. In `frontend/src/lib/components/grid/DataController.svelte`:
   - Remove the imports: `createChangeController` from gridChanges and `createHistoryController` from gridHistory
   - Add imports: `getChangeControllerContext` and `getHistoryControllerContext` from gridContext.svelte.ts
   - Replace `const changes = createChangeController()` with `const changes = getChangeControllerContext()`
   - Replace `const history = createHistoryController()` with `const history = getHistoryControllerContext()`
   - This makes DataController read the SAME instances that GridOverlays populates — commit/discard now operate on the real data.

**IMPORTANT:** GridOverlays is rendered as a child INSIDE GridContainer (`<GridOverlays />` at line 178 of GridContainer.svelte), and DataController is a sibling of GridContainer in the component tree. For `setChangeControllerContext` in GridOverlays to be readable by DataController via `getChangeControllerContext`, DataController must be rendered AFTER GridOverlays in the component tree. Check the render order in the parent component (likely +page.svelte or the GridContextProvider equivalent). If DataController is rendered BEFORE GridContainer (which contains GridOverlays), the context will not be set yet.

**If render order is wrong:** Move the `createChangeController()` and `createHistoryController()` calls and the `set*ControllerContext()` calls UP to the component that is the common ancestor of BOTH GridOverlays and DataController. Looking at the codebase, this would be wherever both `<GridContainer />` and `<DataController />` are rendered — likely the routes `+page.svelte`. In that case:
- The parent component creates the instances and calls `set*ControllerContext()`
- Both GridOverlays and DataController call `get*ControllerContext()` to read the shared instance
- Remove `createChangeController()` and `createHistoryController()` from BOTH GridOverlays and DataController

**Practical approach:** Read the routes `+page.svelte` (or equivalent layout) to determine render order. The safest fix is to create instances in the common ancestor. Since `+page.svelte` currently renders `<GridContextProvider>` which wraps everything, and GridContextProvider is where all `set*Context()` calls happen, the controller instances should be created there too. However, `createChangeController()` calls `getValidationContext()` and `getChangeContext()` internally — so it must run inside a component where those contexts are already set. GridContextProvider sets those contexts, so the controllers can be created right after the context setters in GridContextProvider, OR in a child component that renders before both GridOverlays and DataController.

**Simplest correct approach:** Create instances in GridOverlays (already does this) AND pass them down via context. DataController reads from context. This works as long as GridOverlays renders before DataController accesses the controllers. Since DataController's commit/discard are called by user action (button click), not during initialization, the context will be set by then — GridOverlays mounts before any user can click commit.

**Problem:** When a user clicks away from an active edit (the most common save pattern), two things race:
1. GridContainer `onmousedown` (line 117) calls `edit.save(dataCtx.assets)` — this sets `editCtx.isEditing = false` synchronously
2. FloatingEditor `handleBlur` fires a `setTimeout(() => { if (editCtx.isEditing) ... }, 0)` — by the time the timeout fires, `isEditing` is already false, so the save's `.then(change => onSave?.(change))` never runs, meaning history is never recorded

**Fix part A — GridContainer.svelte:**

In the `onmousedown` handler (around line 110-123), change the `editCtx.isEditing` branch:

BEFORE:
```typescript
if (editCtx.isEditing) {
  edit.save(dataCtx.assets);
  selection.selectCell(row, col);
  return;
}
```

AFTER:
```typescript
if (editCtx.isEditing) {
  // Do NOT call edit.save() here — let FloatingEditor's handleBlur own the save.
  // Just move selection; the blur event on the textarea handles saving + history recording.
  selection.selectCell(row, col);
  return;
}
```

Remove the `edit.save(dataCtx.assets)` call entirely. The mousedown on a different cell will blur the textarea, which triggers FloatingEditor's `handleBlur`, which calls `edit.save()` with the `.then(change => onSave?.(change))` chain that records history. This is the correct single save path.

Also remove the `createEditController` import and `const edit = createEditController()` line from GridContainer if `edit` is no longer used anywhere else in the file. Check first — `edit` may be used in the `ondblclick` handler (line 135-155). If so, keep the import and instance but only remove the `edit.save()` call from mousedown.

**Fix part B — FloatingEditor.svelte handleBlur:**

The current `handleBlur` checks `editCtx.isEditing` INSIDE the setTimeout. Even with the GridContainer fix above, there could be other edge cases where isEditing gets cleared before the timeout fires. Make the check robust by capturing the editing state BEFORE the setTimeout:

BEFORE (lines 133-143):
```typescript
function handleBlur() {
  if (editCtx.autocomplete) {
    editCtx.autocomplete.clear();
  }
  setTimeout(() => {
    if (editCtx.isEditing) {
      edit.save(dataCtx.assets).then(change => { if (change) onSave?.(change); });
    }
  }, 0);
}
```

AFTER:
```typescript
function handleBlur() {
  if (editCtx.autocomplete) {
    editCtx.autocomplete.clear();
  }
  // Capture editing state NOW — before any other mousedown handler can clear it
  const wasEditing = editCtx.isEditing;
  setTimeout(() => {
    if (wasEditing && editCtx.isEditing) {
      edit.save(dataCtx.assets).then(change => { if (change) onSave?.(change); });
    }
  }, 0);
}
```

Wait — this still checks `editCtx.isEditing` inside the timeout, which is the problem. The correct fix with the GridContainer change (removing edit.save() from mousedown) is:

```typescript
function handleBlur() {
  if (editCtx.autocomplete) {
    editCtx.autocomplete.clear();
  }
  // setTimeout(0) prevents race with dropdown mousedown events (EditDropdown, Autocomplete).
  // Those mousedown handlers call e.stopPropagation() and set values before this fires.
  // With GridContainer no longer calling edit.save(), this is the SOLE blur-save path.
  setTimeout(() => {
    if (editCtx.isEditing) {
      edit.save(dataCtx.assets).then(change => { if (change) onSave?.(change); });
    }
  }, 0);
}
```

The key insight: with `edit.save()` removed from GridContainer mousedown, nothing clears `isEditing` before the blur timeout fires. The textarea blur fires because the click on another cell moves focus. The setTimeout(0) still protects against dropdown mousedown events (where we do NOT want to save because the user is selecting a dropdown item, not clicking away).

So the handleBlur function body stays the same — the fix is entirely in GridContainer (removing the premature edit.save() call).

**Summary of changes:**
1. GridContainer.svelte: Remove `edit.save(dataCtx.assets)` from the `onmousedown` handler's `isEditing` branch. Keep the `selection.selectCell(row, col)` call. If `edit` is only used for this one call, also remove the import and controller creation.
2. FloatingEditor.svelte: No changes needed to handleBlur itself — the race condition is eliminated by the GridContainer fix. The existing setTimeout(0) pattern correctly handles dropdown interactions.
