# Phase 01.2: Interaction Grid Layer - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Gut GridOverlays' interaction logic (keyboard, mouse, selection, panel management, context menu) and move it into GridContainer's domain. GridOverlays becomes a thin overlay-only renderer. This eliminates the need for an invisible interaction grid, `data-*` attribute crawling, and pixel math for cell detection — GridContainer already owns the render loop and knows row/col positions natively.

</domain>

<decisions>
## Implementation Decisions

### Architecture split
- **GridContainer.svelte** — Owns the render loop, scroll, AND all user interaction (mouse events, context menu rendering)
- **gridContainer.svelte.ts** — Companion reactive state: selection state, resize drag state, panel management (`setOpenPanel()`), context menu state
- **gridKeyboardHandler.ts** — Pure keyboard event handler (arrow nav, F2, Escape, Ctrl+C/V/Z/Y)
- **gridContainerHelpers.ts** — Shared pure helper functions (e.g. arrow target calculation, overlay computation helpers)
- **GridOverlays.svelte** — ONLY renders visual overlays: dirty cell overlays, selection overlay, copy overlay, other-user cursors, invalid cell hover tooltip. No event listeners, no interaction logic.

### What moves from GridOverlays to GridContainer
- All mouse handlers: `handleMouseDown`, `handleMouseOver`, `handleDblClick`, `handleContextMenu`
- Keyboard handler: `handleKeyDown` (extracted to `gridKeyboardHandler.ts`)
- Selection helpers: `selectCell`, `startSelection`, `extendSelection`, `endSelection`, `resetSelection`
- Panel management: `setOpenPanel()`
- Resize drag state and logic
- Context menu state + `<ContextMenu>` component rendering
- `startCellEdit()` helper
- Window-level event listeners ($effect with mousemove/mouseup/click)

### What stays in GridOverlays
- Dirty cell overlays (`dirtyCellOverlays` derived + rendering)
- Selection overlay rendering
- Copy overlay rendering
- Other-user cursor overlays
- Invalid cell hover tooltip (styled bubble, top-right overlapping the cell, with specific validation message)
- `computeVisualOverlay()` — purely about positioning visual overlays
- All overlay pixel math stays here (this is the one place that needs absolute positioning calculations)

### Dirty cell interaction
- Dirty cells must feel like normal cells — click to select, double-click/F2 to edit
- Since GridContainer owns interaction and the real grid IS the interaction grid, dirty cells are naturally interactive
- Invalid dirty cells get a styled hover tooltip (small text bubble, top-right, overlapping the cell) showing validation message

### Hover feedback
- Move `hover:bg-blue-100 dark:hover:bg-slate-600` highlight to work through the interaction layer (GridContainer's cells)
- GridRow may become simpler or the hover stays on its cells since GridContainer wraps them — Claude's discretion on exact CSS approach

### No invisible grid
- Original idea (invisible event-target grid mirroring GridRow) is scrapped
- The real grid in GridContainer IS the interaction grid — no duplication needed
- No `data-*` crawling for cell detection — GridContainer knows row/col from render loop context

### Claude's Discretion
- Exact CSS approach for hover feedback (whether GridRow keeps hover or GridContainer cells handle it)
- Helper function naming and exact split between `.svelte.ts` and `helpers.ts`
- How GridOverlays receives overlay position data (props from GridContainer vs reading contexts)
- Invalid cell tooltip exact styling and positioning
- Whether `colBounds()` and `assetIndex()` move to helpers or stay where needed

</decisions>

<specifics>
## Specific Ideas

- The companion `.svelte.ts` pattern is already established in the codebase (EditHandler, ContextMenu, HeaderMenu all use it)
- "1 file, 1 job" principle from CLAUDE.md — GridOverlays was doing two jobs (interaction + overlays), now each file has one job
- GridContainer already owns the cell render loop (`{#each visibleData.items as asset}`) — it naturally knows which row/col the user interacts with

</specifics>
