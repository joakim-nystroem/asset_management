# Phase 01.2: Interaction Grid Layer - Research

**Researched:** 2026-03-04
**Domain:** Svelte 5 component refactoring — event handler migration, companion `.svelte.ts` pattern, overlay architecture
**Confidence:** HIGH

## Summary

This phase is a structural refactoring: move all user interaction (mouse, keyboard, selection, panel management, context menu) out of `GridOverlays.svelte` and into `GridContainer.svelte` + companion files. The code is already written and working — the task is decomposition and relocation, not invention.

GridOverlays currently does two jobs: it is both the event-handling layer AND the visual overlay renderer. The "1 file, 1 job" principle from CLAUDE.md requires splitting these. After the refactor, GridOverlays becomes a pure overlay renderer (dirty cells, selection highlight, copy highlight, user cursors) and GridContainer becomes the interaction owner (mouse/keyboard events, cell detection, panel management, context menu).

The companion `.svelte.ts` pattern is already proven in the codebase: `EditHandler`, `ContextMenu`, `HeaderMenu`, and `FilterPanel` all split reactive state / pure logic into a `.svelte.ts` file that the `.svelte` component imports. The same pattern is prescribed for `gridContainer.svelte.ts`.

**Primary recommendation:** Extract interaction logic from GridOverlays into GridContainer and its companion files. The DOM structure will invert — GridContainer renders the row loop directly (no longer inside GridOverlays), and GridOverlays becomes a child/sibling receiving overlay position data as props.

### Pattern 1: Companion `.svelte.ts` for Reactive State

**What:** The `.svelte` component imports reactive state and functions from a sibling `.svelte.ts` file using Svelte 5 runes. The `.svelte.ts` file uses `$state` at module level (or inside factory functions).

**When to use:** When a component has significant reactive state that would clutter the `<script>` block.

**Example (existing — ContextMenu pattern):**
```typescript
// contextMenu.svelte.ts — reactive state, no Svelte component lifecycle needed
export class ContextMenuState {
  x = $state(0);
  y = $state(0);
  visible = $state(false);
  // ...
}
```

**Applied to gridContainer.svelte.ts:**
```typescript
// gridContainer.svelte.ts
// Source: pattern from contextMenu.svelte.ts, editHandler.svelte.ts in this codebase

export type PanelName = 'contextMenu' | 'headerMenu' | 'filterPanel';

// Selection state — mirrors current GridOverlays local vars
export let isDragging = $state(false);

// Resize drag state
export let resizeDrag = $state<{ key: string; startX: number; startWidth: number } | null>(null);

// Context menu position state
export let ctxMenuPos = $state({ x: 0, y: 0, row: -1, col: -1, value: '', key: '' });
```

**Note:** In Svelte 5, `$state` in `.svelte.ts` files works at module scope (singleton) or inside classes/functions. The codebase uses both — `editHandler.svelte.ts` exports pure functions; `contextMenu.svelte.ts` uses a class with `$state` fields.

### Pattern 3: GridContainer Owns Render Loop + Event Listeners

**What:** GridContainer renders the row loop directly (currently done inside GridOverlays as a snippet). Mouse/keyboard events attach at the top-level scrollable div or the inner content div, detecting row/col from the loop's closure context via `data-row`/`data-col` attributes on GridRow cells.

**Current structure:**
```
GridContainer (scroll viewport)
  └── GridOverlays (event div + overlay renderer)
        ├── GridHeader
        ├── row loop → GridRow cells (data-row, data-col)
        └── EditHandler
```

**After refactor:**
```
GridContainer (scroll viewport + event div + row render loop)
  ├── GridHeader
  ├── row loop → GridRow cells (data-row, data-col unchanged)
  ├── GridOverlays (overlay-only renderer, positioned absolute)
  ├── EditHandler
  └── ContextMenu
```

GridContainer retains the `data-*` crawling approach for mouse events — this is the correct approach since GridRow renders the actual cells with those attributes. The "no data-* crawling" statement in CONTEXT.md means no invisible duplicate event grid, not removal of the existing `data-row`/`data-col` detection from real cells.

### Pattern 4: GridOverlays as Props-Driven Overlay Renderer

**What:** GridOverlays becomes a pure renderer. It no longer calls `getContext()` for interaction-related contexts (selCtx, clipCtx etc.) — instead it receives overlay geometry as props, OR it continues reading those contexts directly since it's still inside the context tree.

**Recommendation (Claude's Discretion):** GridOverlays reads contexts directly via `getXContext()` for overlay derivations (`dirtyCellOverlays`, `selectionOverlay`, `copyOverlay`, `otherUserSelections`). This avoids prop-drilling complex computed data. GridOverlays removes all event handlers and interaction state — only retains the `$derived.by()` computations and rendering.

The `computeVisualOverlay()` function and `getWidth()` helper stay in GridOverlays since they are only used for overlay positioning. `assetIndex()` may be duplicated (it exists in both GridOverlays and EditHandler currently) or extracted to `gridContainerHelpers.ts` and imported from there.

### Anti-Patterns to Avoid

- **Calling `getContext()` from `.ts` files:** Svelte context is only accessible in component lifecycle. All context reads MUST stay in `.svelte` files. The keyboard handler and helpers accept contexts as arguments, not via `getContext()`.
- **Putting `$state` in a plain `.ts` file:** Runes only work in `.svelte` or `.svelte.ts` files. `gridKeyboardHandler.ts` and `gridContainerHelpers.ts` are plain `.ts` — no runes.
- **Re-implementing setOpenPanel:** The `setOpenPanel()` pattern is locked from Phase 01.1. Bring the identical implementation to `gridContainer.svelte.ts`; remove from GridOverlays.
- **Breaking the `data-row`/`data-col` contract:** GridRow cells must keep these attributes — event detection in GridContainer depends on them.
- **Attaching events to the scroll container div:** Events should attach to the inner content div (the one with total height), not the scroll viewport, so coordinates align with content space.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reactive module-level state in `.svelte.ts` | Custom observable | Svelte 5 `$state` rune | Already works in `.svelte.ts` files — proven by existing codebase pattern |
| Context access from pure TS | Dependency injection wrapper | Accept contexts as function arguments | Simple and explicit; no magic needed |
| Row/col detection from mouse position | Pixel math against total grid width | `data-row`/`data-col` + `.closest()` | Already established pattern in GridOverlays; works with virtual scroll |
| Event cleanup | Manual removeEventListener tracking | `$effect` return function | Svelte `$effect` cleanup handles this correctly |

**Key insight:** This entire phase is moving existing code between files, not writing new algorithms. The risk is in accidentally losing behavior during the move, not in discovering new patterns.

## Common Pitfalls

### Pitfall 1: `$state` in plain `.ts` file
**What goes wrong:** TypeScript file with `$state()` call fails to compile or produces undefined behavior.
**Why it happens:** Runes are a compile-time transform applied only to `.svelte` and `.svelte.ts` files.
**How to avoid:** `gridKeyboardHandler.ts` and `gridContainerHelpers.ts` are plain `.ts` — no `$state`, no `$derived`, no `$effect`. Only `gridContainer.svelte.ts` can use runes.
**Warning signs:** Import errors mentioning rune transforms; unexpected `undefined` from `$state()`.

### Pitfall 2: Context read outside component tree
**What goes wrong:** `getXContext()` returns undefined at runtime.
**Why it happens:** Svelte context is only available during component initialization. Pure TS files are not components.
**How to avoid:** All `getXContext()` calls stay in `GridContainer.svelte`. Handlers receive context objects as arguments.
**Warning signs:** Runtime error "context not found" or undefined context proxy.

### Pitfall 3: Window event listeners not cleaned up
**What goes wrong:** Multiple listeners stack up on hot-reload or component remount; events fire multiple times.
**Why it happens:** `$effect` runs again on dependency change without proper cleanup.
**How to avoid:** The `$effect` return function removes listeners. Exact pattern already exists in GridOverlays — transplant unchanged.
**Warning signs:** Double-firing events during development; console showing repeated calls.

### Pitfall 4: GridOverlays overlay positions wrong after refactor
**What goes wrong:** Dirty cell overlays, selection overlays are offset incorrectly.
**Why it happens:** `computeVisualOverlay()` depends on `viewCtx.virtualScroll.visibleRange` and `rowHeight`. If GridOverlays is now positioned differently relative to the scroll container, the pixel math breaks.
**How to avoid:** GridOverlays must remain absolutely positioned within the same scrollable content div at the same top offset. Verify `top-8` (32px header) is preserved in the overlay positioning.
**Warning signs:** Overlays appear offset from cells; selection box misaligned.

### Pitfall 5: Hover CSS broken for dirty-cell rows
**What goes wrong:** Hover highlight appears behind dirty cell overlays or doesn't show on hover.
**Why it happens:** Dirty cell overlays are `z-[5]` with `pointer-events-none`. The underlying row div has `hover:bg-blue-50`. If the row structure changes, hover may no longer apply.
**How to avoid:** Keep `hover:bg-blue-50 dark:hover:bg-slate-700` on the row wrapper div in GridContainer's render loop (currently in GridContainer.svelte line 90). GridRow cell-level `hover:bg-blue-100` can be removed or kept — both work since pointer events pass through overlays.
**Warning signs:** Cells show no hover state; hover appears only on some cells.

### Pitfall 6: `isDragging` local var vs reactive state
**What goes wrong:** Selection drag doesn't work — `isDragging` flag isn't reactive across the moved handlers.
**Why it happens:** Currently `isDragging` is a plain `let` variable local to GridOverlays. When moved to `gridContainer.svelte.ts`, it needs to be `$state` if reactivity is needed, or kept as a plain module-level `let` if it's only read/written in the same file's handlers.
**How to avoid:** Since `isDragging` is only checked in `extendSelection()` and `endSelection()`, and those stay in the same file's closure, a plain `let` in `gridContainer.svelte.ts` is sufficient. Do NOT make it `$state` unless a template reads it.
**Warning signs:** Drag selection doesn't extend; `extendSelection` always returns early.

### Pitfall 7: ContextMenu `fixed` vs `absolute` positioning
**What goes wrong:** ContextMenu appears at wrong coordinates after moving to GridContainer.
**Why it happens:** ContextMenu uses `fixed` positioning (top/left from `e.clientX`/`e.clientY`). This is correct and unchanged. The bug would occur only if the `openContextMenu` position calculation is accidentally changed.
**How to avoid:** Transplant `openContextMenu()` and `ctxMenu` state exactly from GridOverlays. ContextMenu renders inside GridContainer's template — since it's `fixed`, its parent doesn't affect positioning.
**Warning signs:** Context menu appears in wrong screen position.

### Current GridOverlays `$effect` for window listeners (move unchanged to GridContainer)
```typescript
// Source: GridOverlays.svelte lines 375-426 (current implementation)
$effect(() => {
  function onMouseMove(e: MouseEvent) {
    if (!resizeDrag) return;
    const delta = e.clientX - resizeDrag.startX;
    const newWidth = Math.max(MIN_COLUMN_WIDTH, resizeDrag.startWidth + delta);
    colWidthCtx.widths.set(resizeDrag.key, newWidth);
  }

  function onMouseUp() {
    endSelection();
    if (resizeDrag) {
      document.body.style.cursor = '';
      resizeDrag = null;
    }
  }

  function onWindowClick(e: MouseEvent) {
    const wasHeaderKey = uiCtx.headerMenu.visible ? uiCtx.headerMenu.activeKey : '';
    const wasFilterOpen = uiCtx.filterPanel.visible;
    setOpenPanel();
    const headerCol = (e.target as HTMLElement).closest('[data-header-col]') as HTMLElement | null;
    if (headerCol) {
      const key = keys[Number(headerCol.dataset.headerCol)];
      if (key !== wasHeaderKey) {
        uiCtx.headerMenu.activeKey = key;
        uiCtx.headerMenu.visible = true;
      }
      return;
    }
    const filterTrigger = (e.target as HTMLElement).closest('[data-filter-trigger]');
    if (filterTrigger && !wasFilterOpen) {
      uiCtx.filterPanel.visible = true;
    }
  }

  window.addEventListener('mousemove', onMouseMove);
  window.addEventListener('mouseup', onMouseUp);
  window.addEventListener('click', onWindowClick);
  return () => {
    window.removeEventListener('mousemove', onMouseMove);
    window.removeEventListener('mouseup', onMouseUp);
    window.removeEventListener('click', onWindowClick);
  };
});
```

### Pure helper — getArrowTarget (move to gridContainerHelpers.ts)
```typescript
// Source: GridOverlays.svelte lines 280-295
export function getArrowTarget(
  key: string,
  current: { row: number; col: number },
  assets: Record<string, any>[],
  colCount: number,
): { row: number; col: number } | null {
  const idx = assets.findIndex((a) => a.id === current.row);
  if (idx === -1) return null;
  const { col } = current;
  switch (key) {
    case 'ArrowUp':    return idx > 0 ? { row: assets[idx - 1].id, col } : null;
    case 'ArrowDown':  return idx < assets.length - 1 ? { row: assets[idx + 1].id, col } : null;
    case 'ArrowLeft':  return col > 0 ? { row: current.row, col: col - 1 } : null;
    case 'ArrowRight': return col < colCount - 1 ? { row: current.row, col: col + 1 } : null;
    default:           return null;
  }
}
```

## Open Questions

1. **GridOverlays positioning relative to GridContainer**
   - What we know: GridOverlays is currently the parent div containing the content. After refactor, GridContainer's inner div owns the content.
   - What's unclear: Should GridOverlays be a sibling div (`position: absolute, inset: 0, pointer-events: none`) rendered alongside the row loop inside GridContainer's inner content div? Or a child below the row loop?
   - Recommendation: Make GridOverlays an `absolute inset-0 pointer-events-none` div as a sibling to the row loop div inside the inner content div. This preserves the same pixel coordinate system.

2. **`assetIndex()` duplication**
   - What we know: This function exists in GridOverlays, EditHandler, and (effectively) multiple files.
   - What's unclear: Should it go to gridContainerHelpers.ts (shared) or stay local in each file?
   - Recommendation: Extract to `gridContainerHelpers.ts` and import from both GridContainer and GridOverlays. Reduces risk of the two implementations diverging.

3. **Invalid cell hover tooltip implementation**
   - What we know: Invalid dirty cells need a tooltip showing validation message (from CONTEXT.md).
   - What's unclear: The `pendingCtx.edits` shape includes `isValid` but NOT a validation message string. The tooltip text source is undefined.
   - Recommendation: During planning, decide if the tooltip just shows "Invalid" or if `PendingContext` needs a `message?: string` field on each edit entry. This is a small scope addition that may be needed.
