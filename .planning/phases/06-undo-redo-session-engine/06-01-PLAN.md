For `onUndo` (currently lines ~72-86), after the `for` loop that calls `changes.update()`, add:

```typescript
// Auto-scroll to first affected cell (Excel behavior)
const firstAction = batch[0];
const row = dataCtx.assets.findIndex(a => a.id === firstAction.id);
const col = colCtx.keys.indexOf(firstAction.key);
if (row !== -1) {
  viewCtx.scrollToRow = row;
  selection.moveTo(row, col !== -1 ? col : 0);

```

For `onRedo` (currently lines ~87-96), after the `for` loop that calls `changes.update()`, add the identical scroll/selection block:
```typescript
// Auto-scroll to first affected cell (Excel behavior)
const firstAction = batch[0];
const row = dataCtx.assets.findIndex(a => a.id === firstAction.id);
const col = colCtx.keys.indexOf(firstAction.key);
if (row !== -1) {
  viewCtx.scrollToRow = row;
  selection.moveTo(row, col !== -1 ? col : 0);
}
```

Key details:
- Use `selection.moveTo()` (NOT `selectCell`) because `moveTo` has no guard — it always updates the selection cursor even if the cell is already selected. This is important for consecutive undo operations on the same cell.
- Use `dataCtx.assets.findIndex(a => a.id === firstAction.id)` to find the row — asset IDs are stable across sorts/filters, row indices are not.
- If `row === -1` (asset filtered out of current view), silently skip scroll/selection. The value revert already happened via `history.undo()` mutating the asset in place.
- For multi-cell batches (paste undo), scroll to the first cell in the batch — this matches Excel behavior.
- `colCtx` and `viewCtx` are already available in the GridOverlays script scope. No new imports needed.
