## User Constraints (from CONTEXT.md)

### Locked Decisions

**Architecture — Three Components**
- **EventListener.svelte** — Dumb orchestrator. Watches all reactive state (URL params, commit clicks, discard, add-row, WS inbound). When anything fires, it creates a typed event and pushes it to the queue. No logic about ordering, errors, or execution.
- **EventQueue.svelte.ts** — Strict serial enforcer. Events go in, events come out one at a time. Next event does NOT start until the current one resolves (success or error). That is its only job.
- **EventHandler.svelte.ts** — Knows how to handle each event type. Commit → fetch POST. Filter → fetch GET. WS delta → patch array. Returns success/error to the queue so it can proceed.

**File Organization**
- All three components live in a dedicated event queue folder (e.g. `src/lib/grid/eventQueue/` or similar)
- Naming convention: `Event*` prefix for all files in this domain

**Queue Rules**
- Single serial FIFO — no parallelism, no priority lanes
- Each event must resolve (success or error) before the next event starts
- Errors surface via toast notification, then queue continues processing the next event
- Queue never stalls on error

**What Goes Through the Queue**
- Rule: if it touches the network, it's a queue event
- All outgoing DB operations (commit/save, filter/fetch, add-row POST, discard)
- All incoming WS operations (realtime deltas from other users)
- Everything post-initial-page-load that involves network I/O

**What Stays Outside the Queue**
- Typing into cells (local array mutation)
- Cell selection and navigation
- Undo/redo history stack operations
- Copy to clipboard
- Any purely local state change

- Whether EventHandler uses a switch/map or separate handler functions per event type
- Folder path naming (`eventQueue/` vs `event-queue/` vs similar)

## Summary

Phase 6.1 introduces a serial event queue to eliminate race conditions in the asset management grid. Currently, all network-touching operations live directly in `DataController.svelte` as bare async functions with no coordination: `commitChanges()`, `addNewRows()`, and the URL-driven search `$effect` can all execute concurrently if triggered in rapid succession. The `handleRealtimeUpdate` callback is registered directly on `realtimeManager` and fires synchronously whenever a WS message arrives — again with no ordering guarantees relative to outgoing network operations.

The architecture decision is sound and well-suited to this codebase. A serial FIFO queue with a Promise-chain enforcer is the canonical pattern for this problem: no external library is needed or appropriate. The queue holds an array of pending events; a single boolean `isProcessing` flag (or a chained Promise) ensures only one handler runs at a time. Events are discriminated union types (TypeScript `type` narrowing). The handler is a large switch/dispatch that calls the existing fetch logic moved from DataController.

The migration strategy is surgical: DataController's async functions (`commitChanges`, `addNewRows`, the URL `$effect` fetch block, `handleRealtimeUpdate`) are extracted into EventHandler. DataController is reduced to a thin EventListener that translates reactive signals into queue pushes. The queue itself is a standalone `.svelte.ts` module with `$state` tracking its running status for optional UI feedback.

**Primary recommendation:** Implement a three-file module under `src/lib/grid/eventQueue/`: `EventQueue.svelte.ts` (Promise-chain serializer), `EventHandler.svelte.ts` (all fetch logic), `EventListener.svelte` (reactive signal → queue push translator). No new npm packages needed.

### Pattern 1: Promise-Chain Serial Queue

**What:** A queue backed by an array and a single chained Promise. Each call to `enqueue()` appends to the chain. Because each handler is `await`ed before the next starts, the chain is strictly serial regardless of how fast events arrive.

**When to use:** Any scenario where events must not overlap. This is the minimal correct implementation — no library needed.

**Example:**
```typescript
// EventQueue.svelte.ts
// Source: standard Promise-chaining pattern (no external library)

type QueuedEvent = {
  event: GridEvent;
  resolve: () => void;
};

function createEventQueue(handler: (event: GridEvent) => Promise<void>) {
  let queue = $state<QueuedEvent[]>([]);
  let isProcessing = $state(false);

  // The chain: each enqueue appends to this promise
  let chain: Promise<void> = Promise.resolve();

  function enqueue(event: GridEvent): void {
    chain = chain.then(async () => {
      isProcessing = true;
      try {
        await handler(event);
      } catch (err) {
        // Errors surface via toast inside handler — queue always continues
        console.error('[EventQueue] Handler threw unexpectedly:', err);
      } finally {
        isProcessing = false;
      }
    });
  }

  return {
    enqueue,
    get isProcessing() { return isProcessing },
  };
}
```

**Key insight:** The `chain = chain.then(...)` pattern means each new event waits for the previous `.then()` to resolve before running. The chain never branches — it grows linearly. Memory usage is O(n) for queued events but events are ephemeral so this is not a concern.

### Pattern 2: Discriminated Union Event Types

**What:** A TypeScript discriminated union where `type` narrows the payload. The handler switch exhaustively covers all event types.

**When to use:** Any time you need type-safe dispatch without runtime type registration overhead.

**Example:**
```typescript
// Part of EventQueue.svelte.ts or a shared types file

// Outgoing DB operations
type CommitEvent   = { type: 'COMMIT' };
type AddRowsEvent  = { type: 'ADD_ROWS' };
type DiscardEvent  = { type: 'DISCARD' };
type FilterEvent   = { type: 'FILTER'; q: string; filters: Filter[]; view: string };
type ViewChangeEvent = { type: 'VIEW_CHANGE'; view: string };

// Incoming WS operations
type WsDeltaEvent  = { type: 'WS_DELTA'; payload: { id: number; key: string; value: any } };

export type GridEvent =
  | CommitEvent
  | AddRowsEvent
  | DiscardEvent
  | FilterEvent
  | ViewChangeEvent
  | WsDeltaEvent;
```

```typescript
// EventHandler.svelte.ts — dispatch switch
async function handleEvent(event: GridEvent): Promise<void> {
  switch (event.type) {
    case 'COMMIT':      return handleCommit();
    case 'ADD_ROWS':    return handleAddRows();
    case 'DISCARD':     handleDiscard(); break;
    case 'FILTER':      return handleFilter(event.q, event.filters, event.view);
    case 'VIEW_CHANGE': return handleViewChange(event.view);
    case 'WS_DELTA':    handleWsDelta(event.payload); break;
  }
}
```

TypeScript will error if a new `GridEvent` type is added but not handled in the switch (with `noImplicitReturns` or explicit exhaustive check).

### Pattern 3: EventListener — Reactive Signal to Queue Push

**What:** A renderless `.svelte` component that uses `$effect` to observe all reactive signals and translates them into queue pushes. It holds no logic about what happens to the event.

**When to use:** This is the "dumb orchestrator" pattern from the CONTEXT.md. The key constraint is that `$effect` in Svelte 5 tracks all reactive reads inside it — so the listener can watch URL params, context state, and WS handler registration all at once.

**Example:**
```svelte
<!-- EventListener.svelte -->
<script lang="ts">
  import { getDataContext, getUiContext } from '$lib/context/gridContext.svelte.ts';
  import type { EventQueue } from './EventQueue.svelte.ts';

  type Props = { queue: EventQueue };
  let { queue }: Props = $props();

  const dataCtx = getDataContext();

  // Commit button click → queue push
  $effect(() => {
    dataCtx.commit = () => queue.enqueue({ type: 'COMMIT' });
  });

  // WS inbound delta → queue push
  $effect(() => {
    realtime.setAssetUpdateHandler((payload) =>
      queue.enqueue({ type: 'WS_DELTA', payload })
    );
  });

  // URL-driven filter/search (replaces the big $effect in DataController)
  $effect(() => {
    const q = reactiveUrl.searchParams.get('q') || '';
    // ... parse filters and view ...
    if (!skipInitialFetch) {
      queue.enqueue({ type: 'FILTER', q, filters, view });
    }
  });
</script>
```

**Critical detail:** The `skipInitialFetch` guard from DataController must migrate into EventListener to avoid a spurious fetch on mount.

### Pattern 4: Error Containment — Never Stall

**What:** The handler catches its own errors and surfaces them via `toastState`. The queue's `.then()` chain always proceeds because the handler never rethrows.

**When to use:** Always — this is a locked queue rule.

**Example:**
```typescript
// EventHandler.svelte.ts
async function handleCommit(): Promise<void> {
  try {
    const response = await fetch('/api/update', { ... });
    if (!response.ok) {
      toastState.addToast('Failed to commit changes.', 'error');
      return; // queue continues — no throw
    }
    // success path
    toastState.addToast(`${n} changes committed.`, 'success');
  } catch (err) {
    toastState.addToast('Network error while committing.', 'error');
    // no rethrow — queue continues
  }
}
```

### Anti-Patterns to Avoid

- **Throwing from EventHandler:** Any uncaught throw in a `.then()` handler will poison the chain and stall the queue permanently. All network errors MUST be caught and converted to toasts.
- **Calling fetch directly from EventListener:** EventListener must only call `queue.enqueue(event)` — never `await fetch(...)`. Mixing logic into the listener defeats the separation of concerns.
- **Re-registering queue on every render:** The `EventQueue` instance must be created once (either in the parent page or passed as a prop) — not created inside a `$effect` or re-created on each render.
- **Bypassing the queue for "quick" operations:** Discard is synchronous but still goes through the queue because future network-touching variants of discard (e.g. server-side revert) must respect ordering.
- **Two queues:** There must be exactly one queue instance. Having DataController AND EventListener both enqueue to separate instances reintroduces race conditions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Promise chain serialization | Custom mutex/lock with Atomics | Native Promise `.then()` chaining | Promise chain IS the mutex — simpler, no shared state issues |
| Event bus with subscribers | EventEmitter or pub/sub system | Direct `queue.enqueue()` call | No fan-out needed; one handler, one queue |
| Retry logic | Custom retry loop in queue | Not needed (yet) | CONTEXT.md says queue continues on error — retry is out of scope |

**Key insight:** The Promise chain pattern is effectively zero additional code — it's the natural shape of sequential async work. Fighting it with locks, semaphores, or libraries would add complexity with no benefit.

## Common Pitfalls

### Pitfall 1: Chain Poisoning on Uncaught Throw
**What goes wrong:** If `handleEvent()` throws an uncaught error, the `.then()` chain transitions to a rejected state. All subsequent `.then()` calls are skipped — the queue is permanently stalled.
**Why it happens:** Promise rejection propagates through `.then()` chains unless a `.catch()` intercepts it.
**How to avoid:** Wrap the entire `handler(event)` call in a try/catch inside the queue's `.then()` body (as shown in Pattern 1). The outer catch in EventQueue is the safety net; the inner catch in each handler is the normal path.
**Warning signs:** Queue stops processing after the first network error. Events pile up but nothing executes.

### Pitfall 2: skipInitialFetch Migration Bug
**What goes wrong:** The `$effect` in DataController that watches `reactiveUrl` has a `skipInitialFetch` flag that prevents a spurious fetch on component mount (the initial URL is already represented in `data.assets` from `+page.server.ts`). If this guard is not replicated in EventListener, every page load triggers an unnecessary fetch.
**Why it happens:** `$effect` runs immediately on mount, so the initial URL state is observed and a FILTER event is pushed before any user action.
**How to avoid:** Migrate the `skipInitialFetch` local variable (initialized to `true`, set to `false` after first effect run) into EventListener's URL-watching `$effect`.
**Warning signs:** Network tab shows a fetch to `/api/assets` immediately on page load with no user interaction.

### Pitfall 3: WS Delta Bypassing the Queue
**What goes wrong:** `handleRealtimeUpdate` in DataController is registered directly with `realtimeManager.setAssetUpdateHandler()`. If this registration is not replaced with a queue-enqueuing wrapper, WS deltas continue to bypass the queue and can interleave with ongoing DB operations.
**Why it happens:** The realtime manager calls its handler synchronously from `ws.onmessage` — it has no awareness of the queue.
**How to avoid:** In EventListener's `$effect`, replace `realtime.setAssetUpdateHandler(handleRealtimeUpdate)` with `realtime.setAssetUpdateHandler((payload) => queue.enqueue({ type: 'WS_DELTA', payload }))`. Clean up in the `$effect` return function.
**Warning signs:** Grid shows stale WS delta values overwriting in-flight commit results.

### Pitfall 4: discard() is Synchronous but Still Needs Queuing
**What goes wrong:** `discardChanges()` in DataController is synchronous (no network call). The temptation is to call it directly without queueing. But if a COMMIT event is in-flight when DISCARD is triggered, the discard runs against mid-commit state and the results are inconsistent.
**Why it happens:** "It's synchronous so it doesn't need to wait" is wrong — it needs to wait for the queue, not for network I/O.
**How to avoid:** DISCARD event goes through the queue like all others. EventHandler's `handleDiscard()` calls the synchronous discard logic but only when the queue dispatches it.
**Warning signs:** Discarding while a commit is in flight leaves the UI in a mixed state (some cells reverted, some committed).

### Pitfall 5: DataContext Action Callbacks Timing
**What goes wrong:** DataController currently registers action callbacks on `dataCtx` in a `$effect`: `dataCtx.commit = commitChanges`. If EventListener registers queue-enqueuing replacements in a separate `$effect`, there is a race between the two effects on mount. One may overwrite the other.
**Why it happens:** Svelte 5 `$effect` execution order is deterministic (same component order, parent before child) but the effect that runs last wins.
**How to avoid:** DataController must stop registering `dataCtx.commit`, `dataCtx.addRows`, etc. EventListener takes over these registrations. Ensure DataController's old registrations are deleted, not left to race.
**Warning signs:** Toolbar buttons call the old direct fetch functions instead of the queue.

### Pitfall 6: EventQueue Created Inside $effect
**What goes wrong:** If the queue instance is created inside a `$effect` or reactive block, it gets recreated on each reactive run, losing all queued events and resetting the Promise chain.
**Why it happens:** Svelte 5 `$effect` re-runs when dependencies change. Any `new` inside it creates a fresh object.
**How to avoid:** Create the queue instance at the top level of a `<script>` block (component initialization), not inside `$effect`. Pass it as a prop to EventListener and EventHandler.
**Warning signs:** In-flight requests are abandoned mid-flight when URL params change.

### Promise-Chain Serial Queue (complete minimal implementation)
```typescript
// EventQueue.svelte.ts
// Pattern: Promise chain as FIFO serializer — no external library

import type { GridEvent } from './EventQueue.svelte.ts';

export type EventQueueInstance = ReturnType<typeof createEventQueue>;

export function createEventQueue(handler: (event: GridEvent) => Promise<void>) {
  let isProcessing = $state(false);

  // The chain grows with each enqueue; each link awaits the previous
  let chain: Promise<void> = Promise.resolve();

  function enqueue(event: GridEvent): void {
    chain = chain.then(async () => {
      isProcessing = true;
      try {
        await handler(event);
      } catch (err) {
        // Safety net: handler should catch internally, but this prevents chain death
        console.error('[EventQueue] Unhandled error in handler:', err);
      } finally {
        isProcessing = false;
      }
    });
  }

  return {
    enqueue,
    get isProcessing() { return isProcessing },
  };
}
```

## Open Questions

1. **Sort and discard: do they need queuing?**
   - What we know: `applySort` is pure in-memory (no network), `discardChanges` is synchronous. The CONTEXT.md rule is "if it touches the network, it's a queue event."
   - What's unclear: Sort toggles `filteredAssets` in place — no queue needed. Discard is synchronous but must respect commit ordering (see Pitfall 4).
   - Recommendation: Sort stays outside the queue (pure local). Discard goes through the queue as a `DISCARD` event so it respects in-flight commits.

2. **`handleAddNewRow` (the local row scaffolding) vs `addNewRows` (the POST)**
   - What we know: `handleAddNewRow` adds a blank row to the local array (no network). `addNewRows` POSTs to `/api/create/asset`.
   - What's unclear: Do both get queue events?
   - Recommendation: `handleAddNewRow` (local scaffolding) stays outside the queue. Only `addNewRows` becomes `ADD_ROWS` event — it is the network operation.

3. **`navigateError` callback: queue or not?**
   - What we know: `navigateError` only calls `selection.selectCell` and `viewCtx.scrollToRow` — pure local state, no network.
   - Recommendation: Stays outside the queue.

4. **Queue backpressure: what if 100 events queue up?**
   - What we know: The Promise chain holds references to all pending events; there is no limit.
   - What's unclear: Is there a realistic scenario where the queue grows unbounded? (Rapid filter typing could enqueue many FILTER events.)
   - Recommendation: For Phase 6.1, no backpressure handling needed. The URL `$effect` already has a `cancelled` flag pattern in DataController — this can be adapted in EventHandler to cancel stale FILTER requests (the latest FILTER event wins, older ones check `cancelled` and return early). This prevents n-concurrent-fetch from becoming n-serial-fetches.

5. **`cancelled` flag for FILTER events**
   - What we know: DataController's URL effect creates a `cancelled` boolean and sets it in the cleanup function to abort stale fetches. This pattern must be preserved.
   - What's unclear: How does the queue interact with the `cancelled` flag? Each queued FILTER event gets its own `cancelled` scope — a new FILTER event does not cancel a previous one that is currently executing (since only one runs at a time anyway).
   - Recommendation: The `cancelled` pattern can be simplified or removed entirely for FILTER events since serial execution means only one FILTER runs at a time. The `return () => { cancelled = true; }` cleanup from `$effect` still fires when the effect reruns — this now just prevents a no-op rather than preventing a race. Keep it for safety.
