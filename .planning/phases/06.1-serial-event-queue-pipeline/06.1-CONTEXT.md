# Phase 6.1: Serial Event Queue Pipeline - Context

**Gathered:** 2026-02-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Serialise all network-touching grid operations through a single event queue so each action completes atomically before the next begins. Local state (typing, cell selection, undo/redo history) stays outside the queue entirely.

</domain>

<decisions>
## Implementation Decisions

### Architecture — Three Components
- **EventListener.svelte** — Dumb orchestrator. Watches all reactive state (URL params, commit clicks, discard, add-row, WS inbound). When anything fires, it creates a typed event and pushes it to the queue. No logic about ordering, errors, or execution.
- **EventQueue.svelte.ts** — Strict serial enforcer. Events go in, events come out one at a time. Next event does NOT start until the current one resolves (success or error). That is its only job.
- **EventHandler.svelte.ts** — Knows how to handle each event type. Commit → fetch POST. Filter → fetch GET. WS delta → patch array. Returns success/error to the queue so it can proceed.

### File Organization
- All three components live in a dedicated event queue folder (e.g. `src/lib/grid/eventQueue/` or similar)
- Naming convention: `Event*` prefix for all files in this domain

### Queue Rules
- Single serial FIFO — no parallelism, no priority lanes
- Each event must resolve (success or error) before the next event starts
- Errors surface via toast notification, then queue continues processing the next event
- Queue never stalls on error

### What Goes Through the Queue
- Rule: **if it touches the network, it's a queue event**
- All outgoing DB operations (commit/save, filter/fetch, add-row POST, discard)
- All incoming WS operations (realtime deltas from other users)
- Everything post-initial-page-load that involves network I/O

### What Stays Outside the Queue
- Typing into cells (local array mutation)
- Cell selection and navigation
- Undo/redo history stack operations
- Copy to clipboard
- Any purely local state change

### Claude's Discretion
- Exact TypeScript event type definitions and discriminated union shape
- Internal queue data structure (array, linked list, etc.)
- Whether EventHandler uses a switch/map or separate handler functions per event type
- Folder path naming (`eventQueue/` vs `event-queue/` vs similar)

</decisions>

<specifics>
## Specific Ideas

- EventListener is intentionally "dumb" — it only translates reactive signals into typed events, never decides ordering or handles errors
- The queue is the single serialization point — nothing bypasses it for network ops
- Paste is local-only (writes to asset array + history), so it does NOT go through the queue

</specifics>
