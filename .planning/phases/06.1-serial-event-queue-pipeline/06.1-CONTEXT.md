# Phase 6.1: Serial Event Queue Pipeline - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace DataController with a 3-file event queue architecture. Serialise all network-touching grid operations through EventListener → EventQueue → EventHandler so each action completes atomically before the next begins. Implement a basic set of event types to get most operations working. Phase 6.2 will do the thorough event-by-event audit.

</domain>

<decisions>
## Implementation Decisions

### Architecture — Three Components Replace DataController
- **EventListener.svelte** — Dumb router. Replaces DataController in the component tree. Watches all reactive state (URL params, commit clicks, discard, WS inbound). When anything fires, creates a typed event object (e.g. `{ type: 'FILTER', ... }`) and pushes it to the queue. No logic about ordering, errors, or execution.
- **EventQueue.svelte.ts** — Strict serial enforcer. Events go in, events come out one at a time. Next event does NOT start until the current one resolves (success or error). That is its only job.
- **EventHandler.svelte.ts** — Dispatch by event type. Sees the type, passes to the matching handler function. Commit → fetch POST. Filter → fetch GET. WS delta → patch array. Returns success/error to the queue so it can proceed.

### DataController Removal
- DataController.svelte is **deleted entirely** — not refactored, not wrapped
- All network logic migrates into EventHandler
- All reactive watching migrates into EventListener
- `+page.svelte` owns asset state ($state for assets/baseAssets) and passes down as props
- Any helpers (e.g. `updateAssetInList`) migrate to appropriate locations

### File Organization
- All three components live in `frontend/src/lib/grid/eventQueue/`
- Naming convention: `Event*` prefix for all files in this domain

### Queue Rules
- Single serial FIFO — no parallelism, no priority lanes
- Each event must resolve (success or error) before the next event starts
- Errors surface via toast notification, then queue continues processing the next event
- Queue never stalls on error

### What Goes Through the Queue
- Rule: **if it touches the network, it's a queue event**
- All outgoing DB operations (commit/save, filter/fetch, discard)
- All incoming WS operations (realtime deltas from other users)
- Outgoing WS broadcasts (selection broadcast to other users) — the local part (selection highlight) is instant, only the network broadcast is queued
- Everything post-initial-page-load that involves network I/O

### What Stays Outside the Queue
- Typing into cells (local array mutation)
- Cell selection and navigation (local — instant)
- Undo/redo history stack operations
- Copy to clipboard
- Paste (local-only: writes to asset array + history)
- Initial page load fetch (initialization, happens before queue starts)
- Any purely local state change

### Optimistic Local + Queued Network Pattern
- Actions that have both a local and network component split into two parts:
  - Local part executes immediately (snappy UX)
  - Network part (WS broadcast, DB write) goes through the queue
- Example: selection is instant locally, WS broadcast to other users is queued

### COMMIT Subsumes ADD_ROWS
- Creating a new row is local-only until committed (temp/scaffold ID)
- COMMIT sends all pending changes (edits + new rows) in one POST
- No separate ADD_ROWS network event needed
- **Future concern (not this phase):** Row ID reservation via WS to prevent duplicate IDs when multiple users create rows simultaneously. Log in backend knowledge base for a future phase.

### Debug Instrumentation
- Add console.log in EventQueue showing each event entering and completing
- Add artificial timing delay (e.g. `await new Promise(r => setTimeout(r, 500))`) so serialization is visually confirmable in devtools
- This is a debugging aid — will be removed or gated behind a flag in a later phase

### Claude's Discretion
- Exact TypeScript event type definitions and discriminated union shape
- Internal queue data structure (array, linked list, Promise chain, etc.)
- Whether EventHandler uses a switch/map or separate handler functions per event type
- How +page.svelte passes asset state down (props, context, or getter/setter pairs)

</decisions>

<specifics>
## Specific Ideas

- EventListener is intentionally "dumb" — it only translates reactive signals into typed events, never decides ordering or handles errors
- The queue is the single serialization point — nothing bypasses it for network ops
- GridContainer is special — it can never be removed from the app and have it keep functioning. It's essentially the app itself.
- The basic event types for this phase: COMMIT, FILTER, DISCARD, VIEW_CHANGE, WS_DELTA (core operations). Phase 6.2 does the full audit of every event type.

</specifics>
