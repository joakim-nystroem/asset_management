## Human Verification Required

The following items require manual testing since they cannot be verified programmatically:

### 1. Race Condition Elimination

**Test:** Open the app, perform a commit (save changes) immediately followed by a filter/search change before the commit completes.
**Expected:** The filter waits for the commit to finish before executing — no interleaved network state. Browser devtools network tab shows sequential (not overlapping) requests. Console shows `[EventQueue] START: COMMIT` ... `[EventQueue] DONE: COMMIT` ... `[EventQueue] START: FILTER` in order.
**Why human:** Race condition elimination requires observing concurrent async behavior in a live browser session with real network timing.

### 2. 500ms Artificial Delay Visibility

**Test:** Trigger any queue event (e.g., click commit or change the search query).
**Expected:** Console shows `[EventQueue] START: {TYPE}`, then approximately 500ms later the fetch fires, then `[EventQueue] DONE: {TYPE}`.
**Why human:** Timing behavior requires a live browser with devtools open.

### 3. Toolbar Action Callbacks End-to-End

**Test:** Perform commit, discard, add new row, and view change from the Toolbar.
**Expected:** All 4 actions function identically to before the migration — same toast messages, same data updates, same UI state transitions.
**Why human:** UI interaction and visual feedback cannot be verified statically.

### 4. WebSocket Delta Serialization

**Test:** Have another browser tab trigger a WS update while a commit is in flight.
**Expected:** The WS_DELTA event queues behind the COMMIT and processes after it completes.
**Why human:** Requires two concurrent browser sessions and real-time WebSocket traffic.
