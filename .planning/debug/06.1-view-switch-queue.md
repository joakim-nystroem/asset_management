hypothesis: View changes are lost because the $effect-to-queue bridge coalesces rapid URL changes — only the LAST URL write fires the effect, and intermediate view changes never get enqueued. Additionally, the `handleViewChange` guard on line 309 compares against stale `viewCtx.activeView` (unchanged until handler runs), blocking switches "back" to current view.

- timestamp: 2026-02-27T12:01
  checked: EventQueue.svelte.ts — the queue itself
  found: Queue implementation is a simple promise-chain FIFO with no deduplication, no rejection logic, no guards. `enqueue()` unconditionally pushes events and chains them.
  implication: The queue itself is NOT the problem. If events reach `enqueue()`, they WILL be queued and processed in order.

- timestamp: 2026-02-27T12:02
  checked: EventListener.svelte lines 306-311 — handleViewChange guard
  found: `function handleViewChange(viewName: string) { if (viewName === viewCtx.activeView) return; updateSearchUrl({ view: viewName }); }`
  implication: Guard compares against `viewCtx.activeView` which is only updated when EventHandler STARTS processing a VIEW_CHANGE (line 227). During rapid clicks before any handler runs, `viewCtx.activeView` remains at its original value. Clicking back to the original view (e.g., 'default' when it's already 'default') is blocked by this guard.

- timestamp: 2026-02-27T12:03
  checked: EventListener.svelte lines 219-248 — URL-driven $effect bridge
  found: The $effect reads `reactiveUrl.searchParams.get('view')` and compares to `untrack(() => viewCtx.activeView)`. It enqueues VIEW_CHANGE only when they differ. The effect is REACTIVE — it runs when reactiveUrl changes, but Svelte 5 effects are batched/deferred to the next microtask.
  implication: If multiple `updateSearchUrl()` calls happen synchronously (rapid clicks), the reactiveUrl is overwritten multiple times, but the $effect only runs ONCE with the FINAL value. Intermediate view changes are lost.

- timestamp: 2026-02-27T12:04
  checked: EventListener.svelte lines 173-188 — updateSearchUrl
  found: `updateSearchUrl` does `reactiveUrl.searchParams.set('view', view)` then `replaceState(new URL(reactiveUrl), {})`. Each call OVERWRITES the previous view param. This is synchronous — no awaiting, no queueing at the URL level.
  implication: Rapid calls to updateSearchUrl overwrite the URL param before the $effect can observe intermediate values.

- timestamp: 2026-02-27T12:05
  checked: EventHandler.svelte.ts line 227 — when activeView is set
  found: `viewCtx.activeView = view` is set at the START of handleViewChange, before the async fetch. This means for the FIRST queued event, activeView updates quickly. But for rapid clicks happening before ANY handler starts, activeView is stale.
  implication: The $effect's `untrack(() => viewCtx.activeView)` comparison uses the stale value, so if the last URL write happens to match the stale activeView, no event is enqueued at all.

- timestamp: 2026-02-27T12:06
  checked: Full trace of default -> ped -> audit -> default rapid scenario
  found: |
    TWO INDEPENDENT ROOT CAUSES:

    ROOT CAUSE 1 — $effect coalescing (lost intermediate views):
    Click "ped": updateSearchUrl sets view='ped'. $effect scheduled but not yet run.
    Click "audit": updateSearchUrl OVERWRITES view='audit'. $effect still pending.
    Click "default": handleViewChange guard blocks (see RC2). Even if it passed, updateSearchUrl would overwrite view='default'.
    $effect finally runs: sees view='audit' (or 'default'), compares to activeView='default'. If 'audit' — enqueues one VIEW_CHANGE. If 'default' — views match, enqueues NOTHING.
    Result: "ped" view change is ALWAYS lost. At most ONE event is enqueued.

    ROOT CAUSE 2 — handleViewChange early-return guard (blocks return-to-current):
    Line 309: `if (viewName === viewCtx.activeView) return;`
    viewCtx.activeView is still 'default' (handler hasn't run yet).
    Clicking 'default' (back to starting view) hits this guard and returns without even touching the URL.
    This means the THIRD click in "default -> ped -> audit -> default" is completely dropped.
  implication: The architecture routes view changes through URL params + reactive $effect as an intermediary, but this introduces lossy coalescing. The queue needs DIRECT enqueue calls, not indirect URL-mediated signaling.

## Resolution

root_cause: |
  Two independent bugs cause rapid view switches to be dropped:

  1. **$effect coalescing** (EventListener.svelte lines 219-248): View changes go through
     `handleViewChange` -> `updateSearchUrl` (modifies reactive URL) -> `$effect` (reads URL, enqueues).
     Svelte 5 effects are batched — if multiple `updateSearchUrl` calls happen synchronously (rapid clicks),
     the $effect only runs once with the FINAL URL value. All intermediate view changes are lost because
     the effect never observes them.

  2. **Stale guard comparison** (EventListener.svelte line 309): `if (viewName === viewCtx.activeView) return;`
     checks against `viewCtx.activeView`, which is only updated when the EventHandler STARTS processing
     a VIEW_CHANGE event (EventHandler.svelte.ts line 227). During rapid clicks before any handler runs,
     `viewCtx.activeView` retains its original value ('default'). Switching back to 'default' hits this
     guard and is silently dropped — the URL is never even updated.

  The fundamental design flaw: using URL params + reactive $effect as the bridge between user action and
  event queue introduces an inherently lossy intermediary. The $effect is appropriate for debounced
  search/filter (where coalescing is desirable), but VIEW_CHANGE is a discrete user action that must
  be queued immediately and directly.

fix: |
  **Recommended fix direction:**
  `handleViewChange` should call `queue.enqueue({ type: 'VIEW_CHANGE', ... })` DIRECTLY instead of
  going through `updateSearchUrl` -> `$effect`. The URL should be updated as a SIDE EFFECT of the
  handler completing (inside EventHandler's handleViewChange), not as the trigger mechanism.

  This preserves the URL-driven pattern for search/filter (where coalescing is correct behavior)
  while ensuring discrete user actions like view switching are never lost.

  Specific changes needed:
  - EventListener.svelte `handleViewChange` (line 308): call `queue.enqueue()` directly
  - EventListener.svelte URL $effect (line 240): remove or skip the VIEW_CHANGE branch
    (the $effect should only handle FILTER events from URL changes)
  - EventHandler.svelte.ts `handleViewChange` (line 224): update URL after fetch completes
    (call the URL update helper or set searchParams directly)
  - Remove the stale guard on line 309 (queue handles serialization, guard is unnecessary)
